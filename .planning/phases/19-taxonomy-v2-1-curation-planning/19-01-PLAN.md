## Target Integrity Validation

### Definição

A validação de integridade de targets será executada como parte de qualquer fase de execução futura:

```
Para cada alias (a → t) em descriptor_aliases.seed.json APÓS curadoria:
  ASSERT t ∈ seed_descriptors ∨ isDocumentedException(a, t)
  
  Se add_target(t) foi executado:
    ASSERT t ∈ seed_descriptors
    ASSERT isPreservedLegacyAlias(a, t) pode ser removido do teste
    
  Se accepted_exception(a, t) foi mantido:
    ASSERT isPreservedLegacyAlias(a, t) permanece no teste
    ASSERT exception documentada em CLOSURE.md
```

### Validação Automatizada

Os testes existentes em `alias_seed_v2.test.ts` já cobrem:

1. Validação estrutural do alias seed JSON
2. Verificação de target existence (com bypass de exceção legada)
3. Rejeição de alias keys que duplicam descritores seed
4. Rejeição de targets candidatos/deferidos para novos aliases
5. Compilação determinística de aliases

Qualquer execução futura deve garantir que **todos os 5 testes passam** com as novas decisões aplicadas.

## Rollback Strategy

### Pré-condições

Antes de qualquer mutação em uma fase de execução futura:

1. **Snapshot:** Copiar `descriptor_aliases.seed.json` para backup rastreável.
2. **Snapshot:** Se `add_target` for executado, copiar `taxonomy-seed.v2.json` para backup.
3. **Baseline metrics:** Registrar contagem de aliases, targets, descritores seed.

### Procedimento de Rollback

1. Restaurar `descriptor_aliases.seed.json` do snapshot.
2. Se `taxonomy-seed.v2.json` foi alterado, restaurar do snapshot.
3. Verificar que `git diff` contra o estado pré-mutação é vazio para os arquivos afetados.
4. Executar `npm run safety:guard` para confirmar que o guard passa.
5. Executar testes de alias (`alias_seed_v2.test.ts`) para confirmar que passam.
6. **NÃO** recompilar artefatos oficiais durante rollback; usar apenas `/tmp` compile se necessário.
