# Research: v2.13 PRD Canonicalization & Core Data Bridge

**Date:** 2026-06-19  
**Source:** `docs/PRD-tecnico.md` (versao 0.2, draft tecnico)

## Summary

O PRD tecnico da Alquem.io ja descreve uma arquitetura-alvo com `PostgreSQL + pgvector + Neo4j + TaxonomySystem`, mas o repositorio atual continua deliberadamente restrito a Layer 1 sem runtime, API, banco ou agentes em execucao. O milestone v2.13 precisa transformar essa tensao em contrato claro: usar o PRD como driver normativo externo sem permitir que o repositorio finja implementar superficies que continuam fora de escopo.

## Findings

### 1. O PRD ja exige shared logical IDs entre Postgres e Neo4j

O item `3.4 Identificadores compartilhados` fixa que uma mesma entidade deve ter o mesmo identificador logico em PostgreSQL e Neo4j. Para o TaxonomySystem, isso implica que o milestone deve documentar como IDs canonicos da taxonomia compilada e do read model podem servir de substrate para futuras entidades centrais, sem depender de IDs internos de banco.

### 2. PostgreSQL e Neo4j possuem ownerships diferentes

O item `3.3 PostgreSQL e Neo4j terao responsabilidades diferentes` ja separa:

- PostgreSQL como fonte principal de dados canonicos/transacionais;
- Neo4j como fonte principal de relacoes, caminhos e travessias.

No contexto deste repositorio, isso exige um bridge que deixe explicito:

- o que a Layer 1 realmente sabe hoje;
- o que e apenas projetavel para um sistema futuro;
- o que permanece derivado vs canonico.

### 3. O PRD menciona capacidades muito alem do escopo atual do repositorio

O PRD cobre chat, agente, tools, RAGgraph, documentos, proveniencia, regras e consultas operacionais. Nada disso deve ser reintroduzido como "escopo implicito" no milestone. O repositorio precisa alinhar linguagem e contratos sem abrir implementacao.

### 4. O read model v2.11/v2.12 e o melhor substrate seguro para planejar Neo4j

O grafo atual ja possui:

- contrato estatico sancionado;
- namespace tipado de IDs;
- validacao sancionada;
- consumer fail-closed;
- envelope de proof estavel;
- fences claras de escopo.

Logo, as regras futuras de projecao Neo4j devem derivar desse substrate, nao de um grafo paralelo imaginado apenas pelo PRD.

### 5. Ha divida documental v2.12 que afeta a ponte para o PRD

Os itens aceitos no fechamento de v2.12 incluem follow-ups como divergencia do help text, doc inaccuracies e readiness baseline ainda incompleta para consumidores futuros. O milestone v2.13 pode reduzir a parte documental/contratual dessa divida, mas sem reabrir runtime nem alteracoes de contrato.

## Planning Implications

- Abrir o milestone com uma fase de canonizacao do PRD e enquadramento de fronteiras.
- Definir o contrato `TaxonomySystem -> PostgreSQL` como ownership/data-bridge, nao como schema SQL.
- Tratar Neo4j como regras futuras de projecao a partir do read model sancionado.
- Fechar com uma fase curta de reducao de divida documental v2.12 e baseline de readiness consolidada.
